# Movie Ticket Booking System — Implementation Plan

## Context
Greenfield repo: only `pom.xml` (Java 17, Maven), `.gitignore`, `.idea/`. We are building a
backend-only, REST + RDBMS movie ticket booking system per the PRD. The hard parts the
evaluation cares about are: **correct serialization of concurrent seat bookings (no
double-allocation)**, **time-bound seat holds that auto-release**, **layered pricing + discount
codes**, **payment / confirmation / refunds under configurable policies**, and **non-blocking
notifications** — all behind **role-based access** (ADMIN, CUSTOMER) with validation, error
handling, and unit + integration tests.

Confirmed decisions:
- **DB:** PostgreSQL for the app, H2 in-memory for tests.
- **Auth:** Stateless JWT with Spring Security; roles `ADMIN` / `CUSTOMER`.
- **Hold expiry:** `@Scheduled` sweeper **plus** lazy expiry check at access time (defense in depth).

Out of scope (per PRD): UI, deployment/containers/CI, microservices, OAuth/SSO/MFA, prod observability.

## Tech Stack
- **Spring Boot 3.3.x** (`spring-boot-starter-parent`), Java 17.
- Starters: `web`, `data-jpa`, `security`, `validation`, plus `postgresql` driver, `h2` (test scope),
  `jjwt` (JWT), Lombok, `spring-boot-starter-test` + `spring-security-test`.
- Schema is generated by Hibernate (`spring.jpa.hibernate.ddl-auto`) — no Flyway. Unique
  constraints / indexes that the concurrency design relies on are declared via JPA annotations
  (`@Table(uniqueConstraints=…)`, `@UniqueConstraint`).
- Build/run: `mvn spring-boot:run` (Postgres via `application.yml`), `mvn test` (H2 profile).

## Core Concurrency Design (the crux)
The bookable unit is a **`ShowSeat`** row — one row per `(show, seat)` created when a show is
published. Status ∈ `AVAILABLE | HELD | BOOKED`.

Serialization strategy (belt **and** suspenders):
1. **Pessimistic locking** — hold/book transitions select the target `ShowSeat` rows with
   `@Lock(LockModeType.PESSIMISTIC_WRITE)` (`SELECT … FOR UPDATE`). Concurrent transactions for the
   same seat block until the first commits, then re-read fresh status and fail fast if no longer
   available. Seats are locked in a deterministic order (by id) to avoid deadlocks.
2. **DB backstop** — exactly one `ShowSeat` row per `(show, seat)` (unique constraint) carries the
   occupancy `status`; a `@Version` optimistic column catches any lost update on a non-locking
   path. A second concurrent attempt re-reads `HELD`/`BOOKED` and fails with a clean 409. (No
   `booking_seat` unique constraint — that would block legitimate rebooking after cancellation.)
3. Hold and booking transitions run in a single `@Transactional` service method (`SERIALIZABLE`
   not required given the row lock; default isolation + `FOR UPDATE` is sufficient and tested).

This is verified by a concurrency integration test (N threads racing for the same seat → exactly
one success, rest 409).

## Data Model (JPA entities)
- **User** — email (unique), passwordHash (BCrypt), role.
- **City** — name (unique).
- **Theater** — city, name, address.
- **Screen** — theater, name (the auditorium that owns a seat layout).
- **Seat** — screen, rowLabel, seatNumber, **seatCategory** (`REGULAR | PREMIUM`); unique
  `(screen, rowLabel, seatNumber)`. This is the *physical* layout.
- **Movie** — title, language, durationMinutes, certification.
- **Show** — screen, movie, startTime, endTime, `showType` (`REGULAR | WEEKEND`), basePrice.
- **ShowSeat** — show, seat, status, price (resolved at publish time), `currentHoldId` (nullable),
  `currentBookingId` (nullable), `@Version`; unique `(show, seat)`. **Single source of truth for
  seat occupancy** — exactly one row per (show, seat); the (show, seat) uniqueness + pessimistic
  lock + @Version *is* the no-double-allocation guarantee. Re-bookable after cancellation.
- **PricingTier** — admin-configurable multipliers/surcharges keyed by `seatCategory` × `showType`
  (e.g. PREMIUM weekend = base × premiumMult × weekendMult). Used to compute `ShowSeat.price`.
- **DiscountCode** — code (unique), type (`PERCENT | FLAT`), value, maxDiscount, minBookingAmount,
  validFrom/validTo, `usageLimit` (0 = unlimited), usedCount, active, `@Version` (guards
  `usageLimit` under concurrent redemption).
- **SeatHold** — user, show, `expiresAt`, status (`ACTIVE | CONVERTED | EXPIRED | RELEASED`); the
  held ShowSeats reference this hold.
- **Booking** — user, show, status (`CONFIRMED | CANCELLED`), subtotal, discountAmount,
  totalAmount, discountCode (nullable), createdAt, `@Version` (guards against double
  cancel/refund).
- **BookingSeat** — booking, showSeat, `pricePaid` (price snapshot); unique `(booking, showSeat)`.
  Historical line item; **no** global unique on `showSeat` so cancelled+rebooked seats retain
  history.
- **Payment** — booking, status (`SUCCESS | FAILED | REFUNDED`), amount, method, gatewayRef
  (mock gateway, deterministic for tests).
- **RefundPolicy** — admin-configurable ordered rules: `hoursBeforeShow >= X → refundPercent`
  (e.g. ≥48h→100%, ≥6h→50%, else→0%).
- **Refund** — booking, amount, status, processedAt.
- **NotificationOutbox** — type, recipient, payload (`text`), status, attempts (async delivery; see
  below).

Schema is generated by Hibernate (`ddl-auto: update` on Postgres, `create-drop` on the H2 test
profile) — no migration tool. Secondary indexes declared via JPA `@Index` on the hot query paths:
`ShowSeat(show_id)`, `ShowSeat(current_hold_id)`, `SeatHold(status, expires_at)`,
`Show(start_time)`, `Booking(user_id)`, `NotificationOutbox(status)`.

**Service-layer invariants** (not enforceable by the schema, validated in services):
a `ShowSeat`'s seat must belong to its show's screen; all `BookingSeat`s share the booking's show;
no overlapping shows on one screen.

## Hold lifecycle & expiry
- `POST /api/holds` → lock target ShowSeats, verify each `AVAILABLE` (or `HELD` by an expired
  hold), set `HELD` + create `SeatHold` with `expiresAt = now + holdTtl` (configurable, default
  ~5 min). Returns hold id + per-seat price breakdown.
- **Sweeper:** `@Scheduled(fixedDelay)` job releases `SeatHold`s past `expiresAt` → seats back to
  `AVAILABLE`, hold `EXPIRED`.
- **Lazy check:** any seat read or hold/book attempt treats a hold past `expiresAt` as released
  before acting, so an expired hold never blocks a new booking even between sweeps.

## Booking, payment, refund flows
- `POST /api/bookings` (body: holdId, optional discountCode, payment method): validate hold is
  `ACTIVE`, owned by caller, not expired; compute subtotal from `ShowSeat.price`, apply discount
  (validate window/limit/min-amount, atomically increment `usedCount`), call **mock payment
  gateway**; on success transition seats `HELD → BOOKED`, create `Booking` + `BookingSeat`s +
  `Payment`, mark hold `CONVERTED`, publish confirmation notification. On payment failure → 402,
  hold left intact for retry.
- `POST /api/bookings/{id}/cancel`: owner-only; look up applicable `RefundPolicy` rule by time to
  show start, compute refund, mark `Booking CANCELLED`, release seats → `AVAILABLE`, create
  `Refund` + mock gateway refund, publish cancellation notification.
- `GET /api/bookings` (caller's history), `GET /api/bookings/{id}`.

## Notifications (non-blocking)
Booking/cancel handlers publish a Spring `ApplicationEvent`; an `@Async` listener (dedicated
`ThreadPoolTaskExecutor`) writes to `NotificationOutbox` and invokes a stub sender (logs the
"email/SMS"). The booking transaction commits without waiting on delivery. A small `@Scheduled`
retry job re-sends failed outbox rows. (Outbox chosen over fire-and-forget so delivery survives a
transient failure and is assertable in tests.)

**Reminder notifications:** a separate `@Scheduled` job scans for upcoming shows within a
configurable lead window and enqueues a reminder per confirmed booking into the same outbox
(de-duplicated so each booking is reminded once). Covers the PRD's "reminder notifications"
alongside confirmation/cancellation.

## REST API surface
**Auth:** `POST /api/auth/register`, `POST /api/auth/login` (→ JWT).
**Admin (ROLE_ADMIN):** CRUD for `/api/admin/cities`, `/theaters`, `/screens`, `/screens/{id}/seats`
(bulk layout), `/movies`, `/shows` (publish → generates ShowSeats), `/pricing-tiers`,
`/discount-codes`, `/refund-policies`.
**Customer (ROLE_CUSTOMER):** `GET /api/cities`, `/theaters`, `/shows` (browse/filter by city,
movie, date), `GET /api/shows/{id}/seats` (live seat map), `POST /api/holds`,
`DELETE /api/holds/{id}` (manual release), `POST /api/bookings`, `POST /api/bookings/{id}/cancel`,
`GET /api/bookings`.

## Cross-cutting
- **RBAC:** Spring Security filter chain validates JWT, populates authorities;
  `@PreAuthorize("hasRole('ADMIN')")` on admin endpoints, ownership checks in service layer for
  customer resources.
- **Validation:** Jakarta Bean Validation on request DTOs (`@NotNull`, `@Email`, `@Future`,
  `@Positive`, etc.).
- **Error handling:** A single global `@RestControllerAdvice` produces a consistent JSON error
  body `{ timestamp, status, error, message, path, fieldErrors }` (`fieldErrors` = field→message
  map for validation). Services throw **custom, meaningful domain exceptions**; the advice
  translates each to one HTTP status — services never set status, controllers never catch.
  - Exception hierarchy under `exception/`: base `BookingDomainException` with subtypes
    `SeatUnavailableException`→409, `HoldExpiredException`→409, `HoldNotFoundException`→404,
    `PaymentFailedException`→402, `ResourceNotFoundException`→404, `DiscountInvalidException`→400,
    `RefundNotAllowedException`→409, `UnauthorizedActionException`→403.
  - Bean Validation failures→400; Spring Security auth/authorization→401/403; any uncaught
    exception→500 (generic message, no stack trace or internals leaked to clients).
  - Exceptions carry context (ids/codes) so the advice can build a clear `message`. No raw
    `RuntimeException`/`IllegalArgumentException` for domain failures.
- **DTO ↔ entity** mapping in service layer (no entity leakage from controllers).

## Key assumptions (to also record in README)
- One screen owns one seat layout; a show binds a movie to a screen+time and generates ShowSeats.
- Pricing resolved & frozen onto `ShowSeat` at show publish; discounts applied at booking time.
- Mock payment/refund gateway (no real PSP); deterministic success in tests, configurable failure
  injection to exercise the 402 path.
- Single-currency, `BigDecimal` money; no tax modeling.
- Hold TTL, sweeper interval, and refund policy are configuration/admin-managed, not hardcoded.
- `DiscountCode.usageLimit == 0` means unlimited uses.
- A show's `showType` (incl. `WEEKEND`) is set explicitly by the admin, not derived from the
  start date's day-of-week.
- `Refund` is single-shot (always recorded `REFUNDED` via the mock gateway); no multi-state refund
  lifecycle.
- `ShowSeat.currentHoldId` / `currentBookingId` are soft `Long` pointers (no DB foreign key) to
  keep the occupancy row decoupled from hold/booking lifecycles.
